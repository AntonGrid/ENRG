# ADR-0004: Device Manifest как локальный источник конфигурации

## Статус
Принято

## Контекст
После регистрации и привязки устройства возникает вопрос: как устройство узнает свои параметры (интервал heartbeat, порог Proof, Trust Level, Capabilities)? Постоянные запросы к серверу создают нагрузку и задержки. Хранение конфигурации только на сервере делает устройство зависимым от сети.

## Решение
После регистрации или изменения политик устройство получает подписанный Device Manifest — компактный JSON-документ, содержащий все необходимые параметры конфигурации. Устройство хранит его локально и использует для работы. Сервер подписывает Manifest своим ключом, устройство проверяет подпись.

Manifest содержит:
- device_id
- trust_level
- capabilities
- heartbeat_interval
- proof_threshold
- policy_version
- oracle_endpoint
- signature

## Обоснование
- Снижение нагрузки на сервер: устройство не запрашивает конфигурацию при каждом запуске.
- Автономность: устройство работает даже при временных проблемах с сетью.
- Безопасность: подписанный Manifest нельзя подделать.
- Гибкость: при изменении политик сервер выдает новый Manifest.

## Последствия
- Устройство должно уметь хранить Manifest во flash-памяти.
- Устройство должно проверять подпись Manifest публичным ключом сервера.
- При изменении конфигурации сервер отправляет новый Manifest через OTA или при следующем heartbeat.
- Устройство сравнивает policy_version с текущим и запрашивает обновление при несовпадении.

## Альтернативы
- Постоянные запросы к серверу — отклонено из-за нагрузки и зависимости от сети.
- Жёстко зашитая конфигурация — отклонено из-за отсутствия гибкости.
